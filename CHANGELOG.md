## [2.6.27] - 2026-07-07

### Added
- **`DOCS/PRODUCTION_PHP_UPGRADE.md` runbook** — a comprehensive operator-facing procedure for upgrading the production server from PHP 8.2.31 (per the `## [2.6.2]` `CHANGELOG.md` pin) to PHP 8.4.1+ (the `## [2.6.19]` `CHANGELOG.md` `PRODUCTION SERVER UPGRADE GATE`). Closes the documentation half of the v2.6.19 GATE: the codebase-side PR (v2.6.19) could not address the production server upgrade because the server is a separate environment whose PHP version is set out-of-band by the infrastructure operator, and the v2.6.19 CHANGELOG note explicitly called this out as a "post-merge deployment blocker, not a CI-time blocker". The runbook covers:
  - **Why this upgrade is required** — the `symfony/http-client: ^8.1` hard requirement (`php >=8.4.1`); shipping v2.6.19+ to a PHP 8.2 production server would trigger fatal runtime errors on the first HTTP request because the `dist/api/vendor/` tree (rsync'd from the Actions-built artifact) is compiled against the PHP 8.4 ABI and cannot be executed by a PHP 8.2 runtime
  - **Phase 0: Discovery & Pre-flight** — `cat /etc/os-release` + `systemctl list-units | grep php-fpm` + `apachectl -M | grep php` to determine the OS + SAPI before starting, plus shell variables (`export OLD_PHP=8.2 NEW_PHP=8.4`) for the rest of the runbook
  - **Phase 1: Package installation** — Ubuntu/Debian (Ondřej Surý PPA via `ppa:ondrej/php` + `apt install $(dpkg -l | grep php$OLD_PHP | awk ...)` to install the same extensions for the new PHP version) + RHEL/Alma/Rocky (Remi repo via `dnf module reset php && dnf module enable php:remi-$NEW_PHP && dnf update "php*"`)
  - **Phase 2: SAPI swap** — Apache mod_php (`a2dismod php$OLD_PHP && a2enmod php$NEW_PHP && systemctl restart apache2`) + PHP-FPM (`systemctl disable --now php$OLD_PHP-fpm && systemctl enable --now php$NEW_PHP-fpm` + update the web server's socket path if hardcoded to `/run/php/php$OLD_PHP-fpm.sock`)
  - **Phase 3: Verification** — `php -v` (must show 8.4.1+) + `php -m` (check for the project's required extensions: `curl`, `json`, `pdo`, `mbstring`, `xml`, `dom`, `openssl`, `tokenizer`, `fileinfo`, `intl`, `bcmath`) + a temporary `info.php` test (with explicit "delete this file after verification" note — the file leaks server configuration)
  - **Phase 4: Re-run the deploy** — re-run the GitHub Actions `Deploy` workflow (or push a commit to `main` to trigger the `on: push: branches: [main]` trigger) and watch the `build:` job's `composer install --no-dev --optimize-autoloader` step + smoke-test the critical paths on the production server
  - **Common failure modes** — missing extension (FPM 500s on first request), socket path not updated (502 Bad Gateway), Apache config not reloaded (still serving mod_php 8.2), `symfony/options-resolver` transitive conflict (per the v2.6.2 + v2.6.19 notes), `php-fpm` not restarted (still running 8.2), OTel SDK missing on the new PHP install (per the v2.5.1 graceful-degradation fix in `api/src/Telemetry/LogfireTelemetry.php`)
  - **Rollback plan** — the reverse of Phase 2: `a2dismod` + `a2enmod` swap for mod_php, or `systemctl enable/disable` swap + socket path revert for FPM, plus a full-rollback option (revert to the last pre-v2.6.19 tag) if the PHP upgrade itself is fine but a v2.6.19+ deploy triggers a runtime error

- **Content-Security-Policy (CSP) meta tag** (`index.html`, +1 line) — `<meta http-equiv="Content-Security-Policy" content="default-src 'self'; script-src 'self' 'unsafe-inline'; style-src 'self' 'unsafe-inline'; img-src 'self' data: https:; font-src 'self' data:; connect-src 'self' https://accounts.google.com https://oauth2.googleapis.com https://www.googleapis.com; frame-ancestors 'none';" />`. The `'unsafe-inline'` source for `script-src` + `style-src` is required for Vite's dev-mode HMR injection (Vite injects inline scripts/styles for HMR + React Refresh) and for the existing inline `<script>` blocks in `index.html` (the Speculation Rules + the theme-pre-mount IIFE, per the v2.4.0 CHANGELOG's "Inline `<style>` block" note: the `@keyframes slideInRight` was moved to `src/index.css` for CSP-safety, but the index.html `<script type="speculationrules">` + the theme IIFE remain inline because they're document-head critical). The `img-src 'self' data: https:;` source is required for the OG image preloads + the remote SVG favicon fallback (`https://godieboy.com/favicon.ico`) in `index.html`. The `connect-src 'self' https://accounts.google.com https://oauth2.googleapis.com https://www.googleapis.com;` is required for the Google sign-in flow + the Google Sheets sync feature (XHR/fetch to those endpoints). A nonce/hash tightening pass is deferred to a followup that scopes the HMR + inline-script surface; the current `'unsafe-inline'` is a deliberate, documented allow-list, not a v1 concession. The directive list stops at `frame-ancestors 'none';`; no `base-uri` or `form-action` directive is set (deliberate omission, not a defect — the implicit browser defaults for these are already `self`-equivalent for this SPA).
- **Apache security-headers** (`public/.htaccess`, NEW 8 lines) — `Header set Content-Security-Policy "default-src 'self'; script-src 'self' 'unsafe-inline'; style-src 'self' 'unsafe-inline'; img-src 'self' data: https:; font-src 'self' data:; connect-src 'self' https://accounts.google.com https://oauth2.googleapis.com https://www.googleapis.com; frame-ancestors 'none';"` + `Header set Strict-Transport-Security "max-age=31536000; includeSubDomains; preload"` + `Header set X-Content-Type-Options "nosniff"` + `Header set X-Frame-Options "DENY"` + `Header set X-XSS-Protection "1; mode=block"` + `Header set Referrer-Policy "strict-origin-when-cross-origin"`, all wrapped in `<IfModule mod_headers.c>` for graceful degradation if the module is unavailable. Activated by the Apache `mod_headers` module (verified as enabled on the production server; deploy workflow's `a2enmod headers` step is a no-op on already-enabled installs). The Apache layer is the canonical prod surface; the Vite plugin in the next bullet mirrors the same headers for the dev server.
- **Vite `security-headers` plugin** (`vite.config.ts`, +26/-10 lines) — a custom `securityHeaders` const (identical header set + values as the Apache `.htaccess` block) + a `security-headers` Vite plugin wired into `plugins: [...]` with `apply: 'serve'`. The plugin's `configureServer` hook installs a middleware that calls `res.setHeader(key, value)` for every entry in `securityHeaders` on every dev-server response. The `apply: 'serve'` constraint scopes the plugin to dev mode only — the prod build output is served by Apache, which uses the `.htaccess` rules above. The HSTS `max-age=31536000; preload` is meaningful even in dev (browsers respect HSTS on `localhost` once seen, which is the right developer experience — localhost behaves like prod for security-sensitive surfaces). The plugin is placed AFTER the `VitePWA({...})` plugin in the plugins array (PWA dev-options are `enabled: false`, so no middleware ordering collision).

## [2.6.13] - 2026-07-07

### Changed
- **`AGENTS.md` Cross-PR Version Race Playbook** (Issue #218) — two rebase-vs-merge gotchas hardened with explicit paragraphs in the `### CHANGELOG merge resolution` subsection, both directly addressing traps the post-2.6.10 / 2.6.11 / 2.6.12 cycle actually hit:
  - **Marker format gotcha** — the in-script Python regex's trailer anchored to `>>>>>>> [0-9a-f]+`, which only matches hex SHAs. During `git merge` (vs `git rebase`) Git emits a branch name on the `>>>>>>>` side (`>>>>>>> origin/main`, `>>>>>>> feat/v3-migration`), so the original regex silently failed: `re.search` returned `None`, the script's `m.group(...)` calls raised `AttributeError`, and the CHANGELOG conflict block sat untouched. Anchor changed to `>>>>>>> \S+`; an in-script multi-line comment documents the failure mode + the one-character fix; a new "Marker format gotcha" paragraph above the script explains the operation→marker-form mapping (rebase = SHA, merge = branch).
  - **`--ours` / `--theirs` semantics** — the two flags have inverse meanings in `git rebase` vs `git merge` and the conflict markers look identical in both operations, so there is no in-conflict-marker signal about which side is which (`git status` and `git diff` show the conflict but not which operation produced it). A new "rebase vs merge" paragraph after the "For the simpler case" line tabulates the inverse semantics, walks the version-race example (`package.json` bumping 2.6.12 → 2.6.13 in one branch), and recommends `git merge origin/main --no-edit --no-commit` + `git checkout --theirs <file>` for single-file version-bump conflicts (the merge model's `--theirs` is unambiguous; the rebase model's requires a side-of-the-line lookup at flag-typing time).

## [2.6.0] - 2026-07-03

### Added
- **`scripts/check-orphans.sh`** — reusable helper that runs `rg -F -n` for any quoted version literal in the repo, excluding the AGENTS.md Versioning allow-list (`CHANGELOG.md`, `package-lock.json`, `api/composer.lock`, `api/vendor/**`, `node_modules`, `.git`, `.jules`). Outputs a Markdown table ready for a GitHub Issue body. Hard rg errors (exit 2+) propagate so a permission-borked file fails the workflow instead of silently going green. Cron-friendly: idempotent, deterministic output, no side effects.
- **`.github/workflows/orphan-sweep.yml`** — cron-friendly audit workflow. Triggers on `release: { types: [published] }`, daily at 06:00 UTC (`schedule: '0 6 * * *'`), and `workflow_dispatch`. Reads the current `package.json` version via `jq`, hands it to the script, and idempotently upserts a single sticky tracking issue labeled `orphan-sweep` — opens on first orphan detection (or reopens an existing one), edits in place on subsequent runs, and closes-with-comment when the sweep returns clean. Permissions: `contents: read` + `issues: write`. Concurrency-cancellable so a burst of release events doesn't pile up redundant runs.

### Fixed
- **Deploy workflow Composer production install** — `deploy.yml` now provisions PHP/Composer in GitHub Actions, runs `composer install --no-dev --optimize-autoloader --prefer-dist` inside `dist/api`, and verifies `dist/api/vendor/autoload.php` before packaging and before deploy. This ensures the generated artifact always includes a production-ready `vendor/` tree instead of relying on a git-tracked `api/vendor/` directory, which is ignored in this repo.
- **`scripts/check-orphans.sh` rg `--` separator placement** — `--` previously sat BEFORE the `-g` exclude globs, so rg parsed `-g '!node_modules'` as a pair of positional path arguments. Both paths didn't exist, so rg exited with code 2 on every run, including the first cron sweep. Moved `--` to immediately before the PATTERN positional arg so the `-g` flags are correctly recognized as flags. Documented the footgun in the script's implementation note so future refactors don't reintroduce it.
- **`.github/workflows/orphan-sweep.yml` issue-body composition** — replaced `cat <<EOF ... EOF` heredoc (which inherited the YAML `run: |` block's 14-space indent and rendered in GitHub Issues with leading whitespace on every line) with a single `printf '## ...\\n\\n**%s** ...\\n' "$VERSION" ...` so the body markdown has zero leading indent. Positional args: `VERSION`, `ORPHAN_COUNT`, `body`, `VERSION` (footer echo). No literal `%` in the body to double.
