## This PR — silent-apply defense

| # | ID | Commit | Title | Files changed |
|---|----|--------|-------|---------------|
| 7 | (new) | (this PR) | Add pre-push hook silent-apply defense (`php -l` on staged .php files) + `dev-loop.md` workflow doc + this follow-up list + tracked installer | `scripts/install-pre-push-hook.sh`, `dev-loop.md`, `DOCS/PHASE_8_FOLLOWUPS.md` |

The **silent-apply defense** is a pre-push hook heuristic that runs
`php -l` on every staged `.php` file and warns on syntax errors. It
defends against the `b4e3ef0` incident class (a `str_replace` edit
silently dropped the trailing `;` from `api/rector.php`'s return
statement; the missing `;` was caught only after PHPStan re-ran the
next day and reported the syntax error). The defense is **local +
advisory only** — the CI quality-gate workflow is the authoritative
layer.

See `dev-loop.md` for the full pre-push hook specification + the
silent-apply defense rationale.

## Forward-looking (Phase 8.1 candidates)

These items would extend the Phase 8 follow-up pattern to additional
defense layers. Listed in priority order (highest impact first). Items
1, 2, 5 are pre-push check extensions (new heuristics that run on
staged files); items 3, 4 are hook meta-changes (behavior or
location).

1. **Suppression-directive rationale check** — extend the pre-push
   hook to detect new `@phpstan-ignore` / `eslint-disable` /
   `react-doctor-disable-next-line` / `@ts-ignore` directives
   **without** a `-- <WHY>` rationale on the same line. The
   `habit-hooks-prompts/non-essential-comment.md` rule already
   enforces this at pre-commit time; a pre-push mirror would be
   defense-in-depth for `git commit --amend` (which can bypass
   the pre-commit hook).
2. **Typed-closure residue check** — extend the pre-push hook with
   a regex check for the untyped `fn ($k) => "..."` pattern in
   staged .php files. Currently enforced by Rector's typed-closure
   rule at CI time. A pre-push mirror would be fast-feedback local.
3. **Strict pre-push mode** — add a `PRE_PUSH_STRICT=1` env var
   that converts the pre-push hook from warn-only to blocking
   (exit 1 on drift or syntax errors). For contributors who want
   the strictest local check.
4. **Tracked `.githooks/` + `core.hooksPath`** — move the pre-push
   hook from `.git/hooks/` (per-clone, not version-controlled) to
   `.githooks/pre-push` (tracked) + `git config core.hooksPath
   .githooks`. Eliminates the need to run
   `scripts/install-pre-push-hook.sh` after every clone.
5. **Pre-push `composer validate --strict`** — extend the pre-push
   hook to run `composer validate --strict --no-check-publish`
   on staged `api/composer.json` and `api/composer.lock` changes.
   Catches composer-validate drift (a `str_replace` on
   `composer.json` that silently drops a constraint, or a stale
   `composer.lock` that no longer matches the manifest) at
   pre-push time. The CI quality-gate workflow already runs
   `composer install --prefer-dist --no-interaction --no-progress`
   on every push, which would surface the drift, but a pre-push
   check gives fast-feedback local. Mirrors the silent-apply
   defense pattern (local + advisory, CI authoritative).
